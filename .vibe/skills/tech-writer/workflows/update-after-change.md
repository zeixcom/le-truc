# Update After Change

## Process

### Step 1: Identify what changed

Run the following to see what has changed since the last release (or as directed):

```sh
git diff main..HEAD -- src/ index.ts examples/
```

If the user describes the change directly, use that description instead of running a diff.

### Step 2: Consult the change-to-document matrix

Read references/document-map.md → Change to Document Matrix. Match the type(s) of change to the affected documents.

### Step 3: Update documents in order

Work through the affected documents in this order (skip any that are unaffected):

1. **JSDoc in `src/`** — update inline documentation first; TypeDoc reads from this
2. **`AGENTS.md`** — update non-obvious behavior entries if behavior changed
3. **`ARCHITECTURE.md`** — update if internal structure changed
4. **Narrative pages in `docs-src/pages/`** — update if public API or patterns changed
5. **`README.md`** — update only if quick-start or installation changed

For each affected document, read the relevant workflow and follow it:

| Document | Workflow |
|---|---|
| `src/` JSDoc | workflows/update-jsdoc.md |
| `AGENTS.md` | workflows/update-agent-docs.md |
| `ARCHITECTURE.md` | workflows/update-architecture.md |
| `docs-src/pages/` | workflows/update-pages.md |
| `README.md` | workflows/update-readme.md |

### Step 4: Do not touch

- `docs-src/api/` — regenerated by TypeDoc on next `bun run build:docs`
- `examples/*/` — use le-truc / le-truc-dev skill for component source files
- `CHANGELOG.md` — use the changelog-keeper skill
