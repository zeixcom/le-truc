# Le Truc v2.0 — Breaking Changes Assessment

> This document catalogues the breaking changes required to ship v2.0, what simplifications become possible when the v1.0 API is dropped, and the migration path for each change. It is an architectural decision document, not a changelog.

---

## Why v2.0?

The v1.1 "factory form" (`defineComponent(name, factory)`) was introduced as an additive improvement while keeping the v1.0 4-param form (`defineComponent(name, props, select, setup)`) alive under a deprecation warning. This backward-compatibility constraint has real costs:

1. **The `Parser<T, U>` and `Reader<T, U>` generics carry a UI object (`U`) that is vestigial in v1.1** — `expose()` always passes `{ host }` as the UI. The generic exists only because v1.0 parsers could receive the full `select` result.
2. **Two overloads for `defineComponent` force defensive branching** throughout `component.ts` — `if (factory) { ... } else { ... }` at every lifecycle point.
3. **The v1.0 effect record pattern** (`Effects<P, U>`, `runEffects`, target-less `on(type, handler)`, target-less `pass(props)`) is a second parallel system with its own type hierarchy that cannot be removed without a major version.
4. **The `fn.length >= 2` fallback in `isParser()`** is a footgun that cannot be removed without a major version, because removing it would silently break any unbranded custom parsers in the wild.
5. **`observedAttributes` behaviour diverges** between the two forms: v1.0 populates it from parsers; v1.1 always sets it to `[]`. Documenting two behaviours for one concept adds ongoing confusion.

Dropping v1.0 in v2.0 lets us resolve all of these cleanly.

A further issue: **v1.1 code already imports from v1.0 deprecated modules**, creating hidden coupling:
- `factory.ts` imports `pass as effectPass` from `src/effects/pass.ts` — the slot-swapping logic lives in a deprecated v1.0 `Effect<P, E>` wrapper
- `effects/pass.ts` imports `Reactive` from `src/effects.ts` — coupling the pass types to the v1.0 `Effect` abstraction
- `dangerouslyBindInnerHTML` is a deprecated `Effect<P, E>` in `effects/html.ts`, but two v1.1 example components call it via the awkward thunk escape hatch: `() => dangerouslyBindInnerHTML(reactive, opts)(host as any, el)`

These are not just dead exports — they are load-bearing code that needs proper v2.0 forms, not just removal.

## Breaking Changes

### BC-1: Remove the 4-param `defineComponent` overload

**Removed:**
```ts
// v1.0 — REMOVED in v2.0
defineComponent<P, U>(name, props, select, setup)
```

**Kept:**
```ts
// v1.1 — the only form in v2.0
defineComponent<P>(name, factory)
```

**Files affected:** `src/component.ts` (remove the `else` branch in `connectedCallback`, remove `attributeChangedCallback`, simplify `static observedAttributes = []` unconditionally, remove `runEffects` call).

**What goes with it:** All types that exist only to type the v1.0 form:

| Removed export | Where defined | Why |
|---|---|---|
| `Component<P>` | `component.ts` | Alias for `HTMLElement & P` — just use the intersection directly |
| `ComponentUI<P, U>` | `component.ts` | UI object for `setup` function |
| `ComponentSetup<P, U>` | `component.ts` | Type of the `setup` function |
| `Initializers<P, U>` | `component.ts` | Drop `U` generic (see BC-3 below) |
| `Effects<P, U>` | `effects.ts` | Return type of `setup` function |
| `ElementEffects<P, E>` | `effects.ts` | Array-or-single Effect union |
| `Effect<P, E>` | `effects.ts` | Target-bound effect function type |
| `runEffects` | `effects.ts` | Effect record executor |
| `UI` | `ui.ts` | Select-return type |
| `ElementFromKey<U, K>` | `ui.ts` | Maps UI key to element type |

`updateElement` and `ElementUpdater` are also removed — they were the shared abstraction underlying the v1.0 `Effect<P, E>` pattern. The v1.1 form composes `EffectDescriptor` functions directly; there is no equivalent concept of a "target-bound effect" in v2.0.

**The entire `src/effects/` subdirectory is removed.** All six v1.0 effect files (`attribute.ts`, `class.ts`, `event.ts`, `html.ts`, `property.ts`, `style.ts`, `text.ts`) go away. `pass.ts` is the only file that contains logic still needed — its slot-swapping implementation moves into `factory.ts` directly (see BC-2). See the "v2.0 File Structure" section below.

### BC-2: Remove v1.0 effect exports and inline `pass` logic into `factory.ts`

**Removed from public API:**

| Export | File | v2.0 replacement |
|---|---|---|
| `on(type, handler, opts?)` | `src/effects/event.ts` | `on(target, type, handler, opts?)` from `FactoryContext` |
| `pass(props)` | `src/effects/pass.ts` | `pass(target, props)` from `FactoryContext` |
| `PassedProp<T, P, E>` | `src/effects/pass.ts` | Stays, moves to `factory.ts` |
| `PassedProps<P, Q>` | `src/effects/pass.ts` | Stays, moves to `factory.ts` |
| `provideContexts(contexts)` | `src/context.ts` | `provideContexts(contexts)` from `FactoryContext` |
| `requestContext(context, fallback)` | `src/context.ts` | `requestContext(context, fallback)` from `FactoryContext` |

**`src/effects/pass.ts` is deleted.** Its slot-swapping logic (the `toSignal` helper and slot `replace`/`restore` loop) moves directly into `makePass` in `factory.ts`. `factory.ts` currently imports `pass as effectPass` from this file and wraps it in `EffectDescriptor` boilerplate — in v2.0 the implementation lives there without indirection.

`PassedProps` and `PassedProp` types move to `factory.ts`. Their dependency on the `Reactive<T, P, E>` union type from `effects.ts` is broken: the `pass` target is always `HTMLElement & Q`, not a generic `Element`. The v2.0 type can use the same union directly (`keyof P | Signal<T> | ((host: HTMLElement & P) => T)`) without the `Reactive` alias from `effects.ts`.

`src/context.ts` keeps `ContextRequestEvent`, `CONTEXT_REQUEST`, and the protocol types (`Context`, `UnknownContext`, `ContextType`, `ContextCallback`) — these are protocol primitives usable outside Le Truc. `provideContexts` and `requestContext` as standalone `Effect`/`Reader` factories are removed.

### BC-3: Simplify `Parser<T>` — drop the `U` generic and the `ui` first argument

**Current signature:**
```ts
type Parser<T, U extends UI> = (ui: U, value: string | null | undefined, old?: string | null) => T
```

**v2.0 signature:**
```ts
type Parser<T> = (value: string | null | undefined) => T
```

**Rationale:**
- In v1.1, `expose()` calls parsers as `parser({ host }, attrValue)` — the `ui` argument is always `{ host }` and none of the built-in parsers use it (they all have `_: UI` as the first parameter).
- The `old` parameter only has meaning in `attributeChangedCallback`, which never fires in v1.1 (and no longer exists in v2.0).
- Dropping `U` eliminates a confusing generic from every call to `asParser()` and every built-in parser factory.

**Files affected:**
- `src/parsers.ts` — update `Parser<T, U>` definition and `asParser<T, U>()` signature
- `src/parsers/boolean.ts`, `string.ts`, `number.ts`, `json.ts`, `date.ts` — remove `_: UI` first param
- `src/component.ts` — update `#initSignals` to call `parser(this.getAttribute(key))` instead of `parser(ui, this.getAttribute(key))`

**Related: remove `LooseReader<T, U>`, simplify `Reader<T, U>`**

`LooseReader` is only used by `read()`, which is removed (see BC-4). `Reader<T, U>` is used as a property initializer in `expose()` — the `u: U` argument is always `{ host: HTMLElement & P }`. In v2.0 this simplifies to:

```ts
// current
type Reader<T, U extends UI> = (ui: U) => T

// v2.0
type Reader<T, P extends ComponentProps> = (host: HTMLElement & P) => T
```

This makes the intent explicit: a Reader receives the host element and computes an initial value from it. The `U` indirection is gone.

`Initializers<P>` drops the `U` generic entirely (no longer parameterised by select result).

---

### BC-4: Remove `read()` utility

**Removed:**
```ts
// v1.0 — REMOVED in v2.0
read(reader, fallback): Reader<T, U>
```

`read()` was a combinator for composing a `LooseReader` with a `Parser` into a typed `Reader`, bridging v1.0's UI-object-based setup with attribute parsing. It was already marked `@deprecated` in v1.1. Without v1.0 and without the `LooseReader` type, the function has no purpose.

**Migration:** Inline the pattern at the call site using a closure:
```ts
// v1.0
expose({ count: read(ui => ui.input.value, asInteger()) })

// v2.0
expose({ count: () => parseInt(input.value, 10) || 0 })
```

### BC-5: Remove `fn.length >= 2` fallback in `isParser()`

**Current behaviour:** `isParser()` checks for `PARSER_BRAND` first; if absent, falls back to `fn.length >= 2` with a `DEV_MODE` warning.

**v2.0 behaviour:** `isParser()` only checks for `PARSER_BRAND`. Unbranded functions are never parsers.

```ts
// v2.0
const isParser = <T>(value: unknown): value is Parser<T> => {
    return isFunction(value) && PARSER_BRAND in value
}
```

**Migration:** All custom parsers must be wrapped with `asParser()`. This has been required since v1.1 was introduced; v2.0 makes it mandatory.

---

### BC-6: Remove `observedAttributes` and `attributeChangedCallback` entirely

**Behavioural change with a clear new mental model.**

In v1.0, properties initialised with a Parser were added to `static observedAttributes`, and `attributeChangedCallback` re-ran the parser whenever an attribute changed on a live document. This created a dual mode: attributes as both initial configuration and as a live reactive channel.

v2.0 draws a hard line:

> **Attributes are for passing initial state from server to client.** The component reads them once at `connectedCallback` and takes over from there. All client-side reactivity goes through the property interface.

`static observedAttributes = []` unconditionally. `attributeChangedCallback` is removed from the `Truc` class. Parsers in `expose()` are called once at connect time — not again. There is no reactive attribute channel in v2.0.

This is not a gap to be filled later — it is a deliberate design decision. The v1.0 attribute-reactive pattern (`host.setAttribute('count', '5')` → `attributeChangedCallback` → signal update) encourages treating attributes as a public API. The v2.0 model is explicit: once connected, drive the component through its property interface (`host.count = 5`). Idiomatic Le Truc components already work this way.

**Migration:** Any component using `host.setAttribute(prop, value)` to drive state from JavaScript should instead use `host[prop] = value`. The attribute-to-property translation at connect time is unchanged.

### BC-7: Replace `dangerouslyBindInnerHTML` with a `WatchHandlers` factory in `helpers.ts`

`dangerouslyBindInnerHTML` is a v1.0 `Effect<P, E>` in `src/effects/html.ts`. Two v1.1 example components use it, but only via the awkward thunk escape hatch noted in its own deprecation comment:

```ts
// current (v1.1 workaround)
() => dangerouslyBindInnerHTML(() => result.get().value ?? '', { allowScripts })(host as any, contentEl)
```

This works but exposes `host as any` and the `Effect<P, E>` calling convention to v1.1 code. The functionality itself (schedule `innerHTML`, optional shadow root, optional script re-execution) is legitimately useful and must survive v2.0.

**v2.0 form:** `dangerouslyBindInnerHTML` becomes a `WatchHandlers<string>` factory in `helpers.ts`, matching the pattern of `bindAttribute` and `bindStyle`:

```ts
// v2.0
const dangerouslyBindInnerHTML = (
    element: Element,
    options?: DangerouslySetInnerHTMLOptions,
): WatchHandlers<string> => ({
    ok: (html) => { schedule(element, () => { /* inject */ }) },
    nil: () => { element.innerHTML = '' },
})
```

Usage in a factory:
```ts
return [
    watch(htmlSignal, dangerouslyBindInnerHTML(contentEl, { allowScripts: true })),
]
```

**Files affected:** `src/effects/html.ts` deleted; implementation and `DangerouslySetInnerHTMLOptions` type move to `src/helpers.ts`.

### BC-8: Remove `Reactive<T, P, E>` from the public API

`Reactive<T, P, E>` is a union type `keyof P | Signal<T> | ((target: E) => T)` defined in `effects.ts`. In v1.0 it was the parameter type for every built-in effect (`setAttribute(name, reactive)`). In v2.0, all v1.0 effects are gone and the type has no public use.

The union concept itself lives on internally — `makePass` in `factory.ts` still resolves property name strings, signals, and functions when injecting values into child components. But it no longer needs the named type, and users of `pass()` never needed to import `Reactive`.

**Removed from public API:** `Reactive<T, P, E>` type. `resolveReactive` function (internal helper used by `updateElement`).

---

### BC-9: Remove `updateElement` and `ElementUpdater`

`updateElement` was the shared abstraction underlying all v1.0-style `Effect<P, E>` functions — it captures a DOM fallback value, then creates a `createEffect` that re-runs the reactive update. This is an internal implementation detail of the built-in DOM effects, not a public composition primitive for v1.1.

**Removed from public API:**
- `updateElement` function
- `ElementUpdater<E, T>` type
- `UpdateOperation` type (internal operation code used by `updateElement` debug logging)

The built-in effects (`setAttribute`, `toggleClass`, `setText`, etc.) continue to use this pattern internally — the implementation stays in `effects.ts`, just unexported.

**Migration:** There is no intended migration path. Any third-party code using `updateElement` directly should be reimplemented as a factory `EffectDescriptor` using `createEffect` from `@zeix/cause-effect` directly.

---

### BC-10: Rename `EventHandlers`/`SensorEventHandler` — resolve the V2 suffix

`src/events.ts` currently exports two parallel sets of types:
- v1.0 form: `EventHandlers<T, U, E>`, `SensorEventHandler<T, Evt, U, E>` (receives `{ event, ui, target, prev }`)
- v1.1 form: `EventHandlersV2<T, E>`, `SensorEventHandlerV2<T, Evt, E>` (receives `{ event, target, prev }`)

In v2.0, the v1.0 forms are removed and `createEventsSensor` is removed entirely — the `createState` + `on` pattern replaces it. The `V2` suffix is dropped from the v1.1 event handler types:

| Old name | v2.0 name |
|---|---|
| `EventHandlersV2<T, E>` | `EventHandlers<T, E>` |
| `SensorEventHandlerV2<T, Evt, E>` | `SensorEventHandler<T, Evt, E>` |

The feature that an event handler can return `{ prop: value }` to batch-update host properties is preserved — this lives in the `on()` helper in `FactoryContext`. No behaviour changes, only naming.

**Migration:** Replace `EventHandlersV2` → `EventHandlers`, `SensorEventHandlerV2` → `SensorEventHandler`. Remove the `ui` field from any v1.0-form sensor handlers. For `createEventsSensor` usage, see the blog post "Removing createEventsSensor".

## v2.0 File Structure

The v2.0 source tree is significantly smaller. The `src/effects/` subdirectory disappears entirely.

```
src/
  component.ts      defineComponent — factory form only, no attributeChangedCallback
  context.ts        Protocol types only (ContextRequestEvent, CONTEXT_REQUEST, Context, etc.)
  effects.ts        EffectDescriptor, FactoryResult, each() — updateElement unexported
  errors.ts         (unchanged)
  events.ts         makeOn, EventHandlers, SensorEventHandler
  factory.ts        makeWatch, makeOn, makePass (with inlined slot-swap), makeProvideContexts,
                    makeRequestContext — plus PassedProps/PassedProp types
  helpers.ts        bind* helpers + dangerouslyBindInnerHTML (moved from effects/html.ts)
  internal.ts       (unchanged)
  parsers.ts        Parser<T>, Reader<T,P>, asParser, asMethod, isParser, isMethodProducer
  parsers/          (unchanged — boolean, date, json, number, string)
  safety.ts         (unchanged)
  scheduler.ts      (unchanged)
  ui.ts             first, all, ElementQueries, ElementFromSelector — no UI/ElementFromKey
  util.ts           (unchanged)
```

**Deleted:**
- `src/effects/attribute.ts`
- `src/effects/class.ts`
- `src/effects/event.ts`
- `src/effects/html.ts` (logic moves to `helpers.ts`)
- `src/effects/pass.ts` (logic moves to `factory.ts`)
- `src/effects/property.ts`
- `src/effects/style.ts`
- `src/effects/text.ts`

## What Becomes Simpler

### `component.ts` reduction

With the v1.0 branch removed, `defineComponent` collapses to a single code path:
- No `isFunction(propsOrFactory)` detection
- No `attributeChangedCallback`
- No `static observedAttributes` conditional
- `#initSignals` is called only from `expose()` with the simplified `Initializers<P>` type
- `#ui` can be typed as `{ host: HTMLElement & P }` directly (no `ComponentUI<P, U>` needed)
- No import of `PassedProps` from `effects/pass.ts` (was only used in the `PassHelper` type, which is now defined against the pass types in `factory.ts`)

Estimated reduction: ~130 lines from `component.ts`.

### `effects.ts` reduction

`runEffects`, `Effects<P, U>`, `ElementEffects<P, E>`, `Effect<P, E>`, `updateElement`, `ElementUpdater`, `UpdateOperation`, and `Reactive` are all removed from the public API. `each`, `EffectDescriptor`, and `FactoryResult` remain. `updateElement` stays as an unexported implementation detail.

### `factory.ts` — self-contained pass implementation

`factory.ts` currently delegates to `effectPass` from `effects/pass.ts`. In v2.0, `makePass` contains the slot-swap logic directly. The file loses one import and gains the pass logic (~50 lines), but the result is easier to follow: the entire `pass()` descriptor lives in one place, alongside `watch()`, `on()`, `provideContexts()`, and `requestContext()`.

### `helpers.ts` — gains `dangerouslyBindInnerHTML`

`dangerouslyBindInnerHTML` moves here as a `WatchHandlers<string>` factory, alongside `bindText`, `bindProperty`, `bindClass`, `bindVisible`, `bindAttribute`, `bindStyle`. All helpers now follow the same `(element, ...opts) => handler` signature shape.

### Simplified generics throughout

`Parser<T>`, `Reader<T, P>`, `Initializers<P>` — all lose their `U extends UI` second generic. This propagates to `asParser`, `asBoolean`, `asString`, `asInteger`, etc. Every call site that currently writes `Parser<string, SomeUI>` becomes `Parser<string>`.

### `parsers.ts` reduction

Remove `LooseReader<T, U>`, `Fallback<T, U>`, `ParserOrFallback<T, U>`, `getFallback`, and `read`.

## Resolved Decisions

**1. `updateElement` and `ElementUpdater`:** Removed from public API (BC-9). They were an abstraction for the v1.0 `Effect<P, E>` pattern. Advanced users build custom effects directly with `createEffect` from `@zeix/cause-effect`. The implementation stays private in `effects.ts`.

**2. `EventHandler` type and `{ prop: value }` return:** The v1.0 `EventHandler` type (from `src/effects/event.ts`) is removed with the `on()` function it typed. The `{ prop: value }` return feature is preserved — it lives in the `on()` helper in `FactoryContext`. The type rename in BC-10 gives the v1.1 sensor handler types clean public names without the `V2` suffix.

**3. `FactoryResult`:** Kept as a named export.

**4. Reactive attributes:** Dropped entirely (BC-6). This is a design decision, not a gap. Attributes configure the component at connect time; the property interface drives all client-side reactivity. No replacement planned for v2.0.

**5. Version number:** `@zeix/le-truc` releases as **2.0**. `@zeix/cause-effect` may increment to 1.1 independently — they are co-developed but do not need matching version numbers.

**6. `dangerouslyBindInnerHTML`:** Survives as a `WatchHandlers<string>` factory in `helpers.ts` (BC-7). Two v1.1 examples use it via a thunk workaround; the v2.0 form is clean.

**7. `src/effects/` directory:** Deleted entirely. `pass` logic inlines into `factory.ts`; `dangerouslyBindInnerHTML` moves to `helpers.ts`; everything else is gone. `effects.ts` (the module) stays but shrinks to `EffectDescriptor`, `FactoryResult`, `each`, and the unexported `updateElement`.
