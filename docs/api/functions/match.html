Remarks#

Async handlers are for external side effects only — DOM mutations, analytics, logging, or any fire-and-forget API call whose result does not need to drive reactive state. Do not call .set() on a signal inside an async handler: use a Task node instead, which receives an AbortSignal, is auto-cancelled on re-run, and integrates cleanly with nil and err branches.

Rejections from async handlers are always routed to err, including rejections from stale runs that were already superseded by a newer signal value. The library cannot cancel external operations it did not start.
